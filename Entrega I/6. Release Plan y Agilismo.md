# **6 Release Plan y Evidencia de Prácticas Ágiles**

## **6.1 Release Plan (Plan de Entregas)**

El desarrollo se estructura en un ciclo de 3 semestres (6 meses). Cada Release está diseñado para entregar valor funcional continuo (MVP evolutivo), coherente con el Product Backlog.

- **Release 1: "Arquitectura Base y Gestión Core" (Semanas 1  8 / Semestre 1**  
  - *Objetivo:* Sistema funcional transaccional sin IA.  
  - *Contenido:* HU-01 (Registro), HU-02 (Login), HU-03 (Crear Ticket), HU-04 (Dashboard y Gestión).  
  - *Hito:* Aplicación web desplegada y Base de Datos configurada con Row Level Security.
- **Release 2: "Automatización y Analítica" (Semanas 9  16 / Semestre 2**  
  - *Objetivo:* Integración de flujos de trabajo y recolección de métricas.  
  - *Contenido:* HU-05 (Notificaciones n8n), Dashboards en tiempo real y armado de Dataset histórico.  
  - *Hito:* Sistema conectado con canales externos (Telegram/Email) y analítica activa.
- **Release 3: "Inteligencia Computacional" (Semanas 17  24 / Semestre 3**  
  - *Objetivo:* Integración de NLP y validación científica.  
  - *Contenido:* HU-06 (Clasificación con Llama-3.1 LLM), Evaluación de Métricas (F1-Score, Matriz de Confusión).  
  - *Hito:* Modelo inteligente integrado y documento académico finalizado.

## **6.2 Evidencia de Prácticas Ágiles (Scrum)**

Dado que el proyecto se desarrolla en modalidad *Solo Developer* (el estudiante asume los roles de Product Owner, Scrum Master y Developer), las ceremonias se adaptan a un marco de auto-gestión y revisión académica:

- **Sprint Planning:**  *Ejecución:* El primer día de cada Sprint de 2 semanas, el estudiante revisa el Backlog en Jira, selecciona las HUs basadas en su capacidad (Story Points) y las mueve a "To Do" en el Sprint Backlog.  
  - *Evidencia:* Sprints iniciados y con metas definidas en Jira Software.
- **Daily Scrum (Adaptado):**  
  - *Ejecución:* Mantenimiento de una bitácora o historial de *commits* en GitHub que reflejan el avance diario y los bloqueos técnicos enfrentados en el código.
- **Sprint Review & Retrospective:**  
  - *Ejecución:* Al final de cada iteración (y formalmente al final de cada semestre), se realiza una demostración del incremento de software funcional al Profesor/Asesor. Tras la demostración, se documentan las oportunidades de mejora (refactorización técnica o ajuste de tiempos) para el siguiente Sprint.  
  - *Evidencia:* Documento de avances semestrales y actas de reunión con el asesor.
